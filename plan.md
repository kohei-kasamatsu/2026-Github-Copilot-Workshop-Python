# 実装計画

## 1. 目的

本ドキュメントは、ポモドーロタイマー Web アプリケーションを段階的に実装するための計画を整理したものである。

対象は [1.pomodoro](1.pomodoro) 配下に構築する Flask + HTML/CSS/JavaScript アプリケーションであり、[architecture.md](architecture.md) と [features.md](features.md) の内容を前提とする。

本計画では、以下を重視する。

- 小さな単位で動作確認しながら進められること
- UI とロジックの責務を早い段階で分離すること
- ユニットテストしやすい構造を維持すること
- MVP を早く成立させ、その後に実用性を高めること

## 2. 実装方針

実装は以下の順で進める。

1. Flask の配信基盤を作る
2. モックに沿った静的 UI を再現する
3. タイマー計算ロジックを pure function として実装する
4. 状態管理を実装する
5. UI と状態管理を接続して動くタイマーにする
6. ポモドーロ固有のモード遷移と進捗管理を入れる
7. 保存と復元を実装する
8. テストを整備する
9. 必要に応じて拡張機能へ進む

各フェーズでは、次のフェーズに進む前に「画面上の成果」または「内部ロジックの成立」が確認できる状態にする。

## 3. フェーズ別実装計画

### フェーズ1. Flask の土台構築

目的:

アプリケーションをブラウザで表示できる最小構成を整える。

実装対象:

- [1.pomodoro/app.py](1.pomodoro/app.py) に Flask のエントリポイントを実装する
- トップページのルーティングを実装する
- [1.pomodoro/templates/index.html](1.pomodoro/templates/index.html) を参照する構成にする
- [1.pomodoro/static/css/style.css](1.pomodoro/static/css/style.css) と JavaScript ファイル群を読み込めるようにする
- [1.pomodoro/config.py](1.pomodoro/config.py) に初期設定値を定義する

完了条件:

- Flask アプリが起動する
- ブラウザでトップページが表示される
- CSS と JavaScript の静的ファイルが配信される
- ルート処理に業務ロジックが混ざっていない

### フェーズ2. 静的 UI の実装

目的:

添付モックに沿った見た目を、まだロジックを入れずに再現する。

実装対象:

- タイトル表示
- 現在モード表示
- 25:00 の残り時間表示
- 円形プログレスの見た目
- 開始ボタンとリセットボタンの見た目
- 今日の進捗カードの見た目
- デスクトップとモバイルのレイアウト調整

主な編集対象:

- [1.pomodoro/templates/index.html](1.pomodoro/templates/index.html)
- [1.pomodoro/static/css/style.css](1.pomodoro/static/css/style.css)

完了条件:

- モックに近い UI が静的表示される
- 主要なレイアウト要素が揃っている
- モバイルでも表示が破綻しない
- タイマー未実装でも画面として成立している

### フェーズ3. タイマー計算ロジックの実装

目的:

画面や保存処理から独立した中核ロジックを先に完成させる。

実装対象:

- 総時間と終了予定時刻から残り秒数を求める関数
- 総時間と残り秒数から進捗率を求める関数
- 残り時間を mm:ss に変換する関数
- 次モード判定関数
- 長休憩判定関数
- 日次集計更新関数の土台

主な編集対象:

- [1.pomodoro/static/js/timer-core.js](1.pomodoro/static/js/timer-core.js)

完了条件:

- DOM に依存しない pure function として実装されている
- 境界値を含む入力を与えても計算が安定する
- タイマーの基本計算が UI なしで成立している

### フェーズ4. 状態管理の実装

目的:

タイマー操作をイベント駆動で扱えるようにし、UI からロジックを切り離す。

実装対象:

- state の定義
- START イベントの適用
- PAUSE イベントの適用
- RESET イベントの適用
- TICK イベントの適用
- COMPLETE イベントの適用
- timer-core.js を用いた状態更新
- 現在時刻取得を差し替え可能にする設計

主な編集対象:

- [1.pomodoro/static/js/timer-store.js](1.pomodoro/static/js/timer-store.js)

完了条件:

- UI がなくても状態遷移を追える
- 開始、停止、リセット、完了の挙動が store 単体で成立する
- 状態管理が DOM と localStorage に依存していない

### フェーズ5. UI とタイマーの接続

目的:

ここまでで作った UI とロジックを接続し、実際に動くタイマーを完成させる。

実装対象:

- DOM 要素の取得
- state を受け取って画面へ反映する描画処理
- 開始ボタンとリセットボタンのイベント接続
- 実行中と停止中の UI 表示切り替え
- 円形プログレスの動的更新
- setInterval から TICK を送る処理

主な編集対象:

- [1.pomodoro/static/js/timer-ui.js](1.pomodoro/static/js/timer-ui.js)
- [1.pomodoro/templates/index.html](1.pomodoro/templates/index.html)

完了条件:

- 開始するとタイマーが進行する
- 一時停止と再開ができる
- リセットで初期状態に戻る
- 残り時間表示とリング表示が同期する
- UI 層に計算ロジックが入り込んでいない

### フェーズ6. ポモドーロ固有機能の実装

目的:

単なるカウントダウンではなく、ポモドーロアプリとして必要な挙動を完成させる。

実装対象:

- 作業完了後に短休憩へ遷移する
- 規定回数ごとに長休憩へ遷移する
- 休憩完了後に次の作業へ戻る
- 今日の完了回数を更新する
- 今日の集中時間を更新する
- 進捗カードへ反映する

主な編集対象:

- [1.pomodoro/static/js/timer-core.js](1.pomodoro/static/js/timer-core.js)
- [1.pomodoro/static/js/timer-store.js](1.pomodoro/static/js/timer-store.js)
- [1.pomodoro/static/js/timer-ui.js](1.pomodoro/static/js/timer-ui.js)

完了条件:

- 1 セッション完了後に次のモードへ自動遷移する
- 長休憩条件が正しく機能する
- 今日の進捗が画面に反映される

### フェーズ7. 状態保存と復元の実装

目的:

リロードに耐える実用的なタイマーにする。

実装対象:

- localStorage 保存処理
- localStorage 読み込み処理
- 保存キー管理
- 不正データへの防御
- ページ読み込み時の状態復元
- 期限切れタイマー状態の補正
- 日付単位の進捗保存

主な編集対象:

- [1.pomodoro/static/js/storage.js](1.pomodoro/static/js/storage.js)
- [1.pomodoro/static/js/timer-store.js](1.pomodoro/static/js/timer-store.js)

完了条件:

- ページ再読み込み後にタイマー状態が復元される
- 今日の進捗が保持される
- 壊れた保存データがあってもアプリが落ちない

### フェーズ8. テストの整備

目的:

主要ロジックを自動テストで担保し、以後の拡張で壊れにくくする。

実装対象:

- Flask のトップページ表示テスト
- timer-core.js のユニットテスト
- timer-store.js の状態遷移テスト
- 進捗集計ロジックのテスト
- 必要に応じて storage.js の異常系テスト

主な編集対象:

- [1.pomodoro/tests/test_app.py](1.pomodoro/tests/test_app.py)
- [1.pomodoro/tests/test_pomodoro_service.py](1.pomodoro/tests/test_pomodoro_service.py)
- JavaScript テストファイル群

完了条件:

- 主要ロジックのテストが自動実行できる
- 境界値と代表的な遷移がカバーされる
- 今後の修正で回帰を検知できる状態になっている

### フェーズ9. 拡張機能の実装

目的:

MVP 完了後に、必要な機能を優先度順に追加する。

候補機能:

- 作業時間や休憩時間の設定変更
- 通知音やブラウザ通知
- 週次、月次統計
- 履歴表示
- Flask API の追加
- SQLite 永続化

完了条件:

- MVP を壊さずに拡張できている
- 追加機能が既存の責務分離を壊していない

## 4. 推奨する実装単位

1 回の実装で広げすぎないために、各フェーズ内でも以下の粒度を基本とする。

### 4.1 小さく分ける対象

- 画面構造
- スタイル調整
- pure function 単位のロジック
- 状態遷移イベント単位の処理
- 保存と復元の入口処理
- テストケース単位の追加

### 4.2 避けるべき実装粒度

以下を同時に 1 ファイルへ詰め込まない。

- タイマー計算
- DOM 書き換え
- setInterval 制御
- localStorage 保存
- モード遷移判定

この粒度で実装すると、動いてもテストしにくくなり、後からの修正が重くなる。

## 5. マイルストーン

### マイルストーン1

フェーズ1 とフェーズ2 を完了し、静的な画面がブラウザで見える状態にする。

### マイルストーン2

フェーズ3 からフェーズ5 を完了し、開始、停止、リセットができる動くタイマーを完成させる。

### マイルストーン3

フェーズ6 とフェーズ7 を完了し、ポモドーロ遷移と状態復元を含む MVP を完成させる。

### マイルストーン4

フェーズ8 を完了し、主要機能を自動テストで担保する。

### マイルストーン5

フェーズ9 に進み、必要な拡張機能を追加する。

## 6. 最初に着手すべき範囲

最初の開発サイクルでは、以下までを目標とするのが妥当である。

1. フェーズ1 の完了
2. フェーズ2 の完了
3. フェーズ3 の一部完了

この時点で、Flask 上でモックに近い画面が表示され、タイマーの基本計算ロジックの実装が始まっている状態を作る。その後、フェーズ4 とフェーズ5 で動くタイマーに進む。

## 7. 結論

本アプリケーションは、UI、ロジック、状態管理、保存処理を明確に分けながら、段階的に機能を積み上げる進め方が適している。

特に、以下の順序を守ることが重要である。

1. 配信基盤
2. 静的 UI
3. pure function のロジック
4. 状態管理
5. UI 接続
6. ポモドーロ固有機能
7. 保存と復元
8. テスト

この順で進めることで、各段階で成果を確認しながら、安全に MVP まで到達できる。
