# Webアプリケーションアーキテクチャ案

## 1. 目的

本プロジェクトでは、[1.pomodoro](1.pomodoro) 配下に Flask と HTML/CSS/JavaScript を用いたポモドーロタイマーの Web アプリケーションを構築する。

UI はモックをベースにしつつ、以下を満たす構成を採用する。

- 実装を段階的に進めやすいこと
- タイマー処理を安定して動かせること
- ユニットテストしやすいこと
- 将来的な機能追加に耐えられること

## 2. 全体方針

アプリケーションは、Flask を画面配信と将来の API 提供に使い、タイマーの状態管理とカウントダウンはブラウザ側で実行する。

理由は以下の通り。

- 秒単位で更新されるタイマーを毎秒サーバーに依存させる必要がない
- ブラウザ側で完結した方が体感速度と実装効率がよい
- localStorage を使えばリロード後の状態復元がしやすい
- Flask 側はシンプルに保ち、必要になった段階で API や永続化を拡張できる

## 3. アーキテクチャ概要

責務は以下の 4 層に分ける。

### 3.1 Flask 層

責務:

- HTML テンプレートを返す
- 必要に応じて初期設定値を注入する
- 将来的に統計情報やセッション保存 API を提供する

MVP 段階では、まずトップページ表示を担う最小構成で開始する。

### 3.2 View 層

責務:

- 画面の骨組みを HTML で定義する
- モックに沿ったレイアウトと視覚表現を CSS で実装する

主要 UI:

- タイトル
- 現在のモード表示
- 円形プログレス付きタイマー
- 開始、一時停止、リセット操作
- 今日の進捗カード

### 3.3 フロントエンドロジック層

責務:

- タイマー状態の保持
- 残り時間と進捗率の計算
- モード遷移の制御
- localStorage への保存と復元

この層は DOM 操作からできるだけ分離する。

### 3.4 永続化層

初期段階ではブラウザの localStorage を使用する。

保存対象:

- 実行中タイマーの状態
- 終了予定時刻
- 今日の完了回数
- 今日の集中時間

将来的に複数端末間同期や履歴分析が必要になった場合は、Flask + SQLite に移行または併用する。

## 5. 各ファイルの責務

### 5.1 [1.pomodoro/app.py](1.pomodoro/app.py)

- Flask アプリケーションのエントリポイント
- ルーティング登録
- テンプレート描画
- 将来的な API エンドポイント定義

ここには複雑な業務ロジックを置かない。

### 5.2 [1.pomodoro/config.py](1.pomodoro/config.py)

- 作業時間
- 短休憩時間
- 長休憩時間
- 長休憩に入るまでの回数

設定値を一元管理する。

### 5.3 [1.pomodoro/services/pomodoro_service.py](1.pomodoro/services/pomodoro_service.py)

- 設定値の解決
- 将来的な統計集計ロジック
- API 追加時のセッション処理

Flask のルートから切り離した業務ロジックを保持する。

### 5.4 [1.pomodoro/templates/index.html](1.pomodoro/templates/index.html)

- タイマー画面の構造定義
- ボタンや表示領域のマークアップ
- Flask から渡された初期設定の埋め込み

### 5.5 [1.pomodoro/static/css/style.css](1.pomodoro/static/css/style.css)

- モックに沿ったレイアウトと装飾
- 円形プログレスの見た目
- ボタン状態の見た目
- モバイル対応

### 5.6 [1.pomodoro/static/js/timer-core.js](1.pomodoro/static/js/timer-core.js)

タイマーの中核ロジックを置く。

- 残り秒数計算
- 進捗率計算
- モード遷移判定
- 長休憩判定
- 集計更新

このファイルは pure function を中心に構成し、DOM や localStorage に依存しないようにする。

### 5.7 [1.pomodoro/static/js/timer-store.js](1.pomodoro/static/js/timer-store.js)

- アプリケーション状態の保持
- イベント適用
- timer-core.js と storage.js の仲介

開始、停止、リセット、完了時処理などの状態遷移を集約する。

### 5.8 [1.pomodoro/static/js/timer-ui.js](1.pomodoro/static/js/timer-ui.js)

- DOM 取得
- 画面描画
- ボタンイベント登録
- プログレスリング更新

ロジック計算は行わず、受け取った state を画面に反映する責務に限定する。

### 5.9 [1.pomodoro/static/js/storage.js](1.pomodoro/static/js/storage.js)

- localStorage の読み書き
- 保存形式の変換
- 日付キー単位の保存管理

ストレージ依存を分離し、テスト時に差し替えやすくする。

## 6. 状態設計

フロントエンドでは、少なくとも以下の状態を保持する。

- mode: work / short_break / long_break
- isRunning: タイマーが動作中かどうか
- durationSeconds: 現在モードの総時間
- endTimestamp: 終了予定時刻
- remainingSeconds: 現在の残り時間
- completedPomodoros: 今日完了した作業回数
- focusSecondsToday: 今日の累積集中時間

重要な点は、毎秒単純に残り秒数を減算するのではなく、現在時刻と endTimestamp の差から残り時間を再計算すること。これにより、setInterval のずれやタブ非アクティブ時の影響を受けにくくなる。

## 7. 状態遷移の考え方

タイマー制御は小さな状態マシンとして扱う。

主なイベント:

- START
- PAUSE
- RESET
- TICK
- COMPLETE

例:

- START: 現在モードでタイマー開始
- PAUSE: 実行中タイマー停止
- RESET: 現在モードの初期時間に戻す
- TICK: 現在時刻から残り時間を再計算する
- COMPLETE: 作業完了時に短休憩または長休憩へ遷移する

この設計により、UI テストではなく状態遷移テストで主要ロジックを検証できる。

## 8. データ保存方針

### 8.1 初期実装

localStorage を採用する。

用途:

- リロード後のタイマー復元
- 今日の進捗表示
- セッション継続性の確保

### 8.2 将来拡張

以下の要件が出た場合は、Flask 側の API と SQLite の導入を検討する。

- 複数端末でデータを共有したい
- 週次、月次の履歴を見たい
- ログイン機能を追加したい
- 完了セッションを長期保存したい

その場合の候補 API:

- GET /api/settings
- GET /api/stats/today
- POST /api/sessions

## 9. ユニットテストしやすくするための設計上の追加方針

ユニットテスト容易性の観点では、以下を明確に守る。

### 9.1 pure function を中心にする

以下は DOM やストレージに依存しない関数として実装する。

- 残り時間計算
- 進捗率計算
- 次モード判定
- 長休憩判定
- 集計更新

### 9.2 DOM 操作を薄く保つ

UI 層は state を受け取って描画するだけにする。タイマー計算やモード遷移を UI に持ち込まない。

### 9.3 storage をアダプタ化する

localStorage に直接アクセスするコードを散在させず、storage.js に集約する。これにより、テストではモックストレージに差し替えられる。

### 9.4 時刻取得を抽象化する

Date.now をロジックのあちこちで直接呼ばず、時刻取得関数や clock オブジェクト経由にする。固定時刻を使った再現性の高いテストが可能になる。

### 9.5 Flask でもルートとロジックを分離する

ルート関数では入出力とレスポンス生成に集中し、集計や判定ロジックは service 層に置く。

## 10. テスト戦略

### 10.1 JavaScript ユニットテスト対象

- 指定時刻に対する残り時間計算
- 進捗率の境界値計算
- 作業完了時のモード遷移
- 4 回ごとの長休憩遷移
- 日次集計の加算と日付切り替え処理

### 10.2 Flask ユニットテスト対象

- トップページが正常に表示されること
- API が正しい JSON を返すこと
- 不正な入力に対して適切なステータスコードを返すこと

### 10.3 テストしにくい構成として避けるべきもの

以下を 1 つの処理に混在させない。

- setInterval 内での状態更新
- localStorage 保存
- DOM 書き換え
- モード遷移判定

これらを分離しないと、ブラウザ環境依存が強くなり、単体テストの価値が下がる。

## 11. 実装ステップ

推奨する実装順は以下の通り。

1. Flask でトップページ配信の土台を作る
2. HTML と CSS でモックのレイアウトを再現する
3. JavaScript でタイマー表示とプログレス更新を実装する
4. 状態マシンとして開始、停止、リセット、完了遷移を実装する
5. localStorage で状態復元と今日の進捗を実装する
6. 必要に応じて API と SQLite へ拡張する

## 12. 結論

本アプリでは、Flask は軽量な配信層として使い、タイマーの実体はブラウザ側に持たせる構成を採用する。さらに、ロジック、UI、永続化を明確に分離することで、モックに沿った UI を実現しつつ、段階的な実装とユニットテストの両立を図る。

特に重要なのは以下の 3 点である。

- タイマーの業務ロジックを pure function として分離すること
- DOM 操作と storage 操作をアダプタ層に閉じ込めること
- Flask 側でもルートとサービスを分離すること

この方針により、MVP を素早く構築しながら、将来の拡張性と保守性を確保できる。
